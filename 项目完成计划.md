# PocketCoder 项目完成计划

## Summary
- 目标是在现有方案文档和 9 个项目专用 skills 的基础上，把 PocketCoder 从“规划与脚手架阶段”推进到“V1 可真实使用最小版”。
- 完成标准不是单个模块可运行，而是主链路一次跑通：桌面启动 `codex` 会话、手机完成配对、手机查看实时输出、手机发送 prompt、手机处理授权、桌面恢复本地输入后远控立即失效。
- 实施顺序固定为：monorepo 基础层 -> 协议层 -> relay -> agentd -> web -> E2EE 配对 -> Windows 控制恢复 -> 自托管部署 -> 测试与验收。
- 所有实现都必须保持高内聚、低耦合：`protocol` 只做共享契约，`relay` 只做路由/注册/回放，`agentd` 只做本地事实来源，`web` 只做移动端交互与浏览器侧能力，部署与测试分别独立收口。

## Implementation Plan

### Phase 1. Monorepo 基础落地
- 用 `typescript-monorepo-bootstrap` 在仓库内生成根层和 4 个 workspace：`apps/web`、`apps/agentd`、`apps/relay`、`packages/protocol`。
- 根目录统一落下 `package.json`、workspace 配置、`tsconfig.base.json`、`eslint.config.mjs`、基础 `.gitignore`、根脚本和 CI 最小入口。
- 根脚本固定至少包含：`dev`、`build`、`lint`、`typecheck`、`test`。
- `infra/`、`设计方案/`、`.codex/skills/` 保持与业务代码解耦，不把部署、文档、AI 资源混入应用源码。

### Phase 2. 协议层完成
- 用 `protocol-schema-design` 实现 `packages/protocol`，作为唯一共享契约源。
- 固定协议公共元素：`protocolVersion`、`messageId`、`timestamp`、`type`、`payload`。
- 固定首版消息类型：配对、设备注册、会话列表、会话订阅、输出增量、状态变化、授权请求/响应、发送 prompt、中断、恢复桌面控制、Ack、Error。
- 固定会话状态枚举：`idle`、`running`、`waiting_input`、`waiting_approval`、`error`、`disconnected`。
- 所有跨端入站消息都通过 Zod 校验；`relay` 与 `agentd` 只消费 `packages/protocol` 导出的 schema 和类型，不允许本地重复定义。

### Phase 3. Relay 服务完成
- 用 `fastify-websocket-relay` 实现 `apps/relay` 的服务骨架。
- 先完成结构层：`transport/http`、`transport/ws`、`security`、`storage`、`modules`、`infra`。
- HTTP 层只负责配对相关入口、健康检查和最小管理入口；WebSocket 层只负责连接、订阅、心跳、转发和回放。
- 存储层先落 SQLite 仓储边界：设备、配对 token、会话路由、心跳、TTL 加密事件 blob、已撤销设备、命令去重记录。
- Relay 业务边界固定：
  - 不运行 AI 会话逻辑。
  - 不持有会话明文。
  - 只保留最小元数据和密文事件。
  - 支持最近窗口回放，不做长期历史归档。
- 对外最小行为固定：
  - 注册桌面与手机设备。
  - 校验 token 和设备访问能力。
  - 路由会话订阅与事件转发。
  - 保证 `messageId` 去重。

### Phase 4. Agentd 桌面端完成
- 用 `node-pty-agentd` 实现 `apps/agentd`。
- 先完成本地骨架：`cli`、`providers/codex`、`sessions`、`transport`、`security`、`platform/windows`、`infra`。
- `agentd` 必须成为桌面端事实来源：
  - 负责启动真实 `codex` PTY。
  - 维护本地会话注册表与状态机。
  - 解析输出并产出结构化事件。
  - 执行来自手机端的 prompt、approval、interrupt、resumeDesktopControl 命令。
- 安全入口先只接到边界，不把深层 E2EE 逻辑散到 provider、session、transport。
- Windows 平台层和通用 agent 运行层严格分离，避免后续控制恢复逻辑污染主会话流。

### Phase 5. Web 移动端完成
- 用 `nextjs-pwa-mobile` 实现 `apps/web`。
- 固定 App Router 页面：`/pair`、`/sessions`、`/sessions/[sessionId]`、`/connection-error`，根页只做入口聚合。
- 页面层只做路由装配、移动端展示和交互壳，不直接承载协议解析、存储细节、WebSocket 连接细节和密码学实现。
- 浏览器侧目录固定：
  - `features/` 管配对、会话、授权、连接场景。
  - `lib/crypto/` 管密钥和解密接入。
  - `lib/realtime/` 管 WS 重连、订阅和分发。
  - `lib/storage/` 管本地设备信息和轻量持久化。
  - `lib/pwa/` 管 manifest、安装和 service worker 接口。
- UI 完成标准：
  - 配对页可以发起配对。
  - 会话列表页能展示当前会话摘要。
  - 会话详情页能展示实时输出、底部 prompt 输入、授权卡片、显式释放控制权入口。
  - 连接异常页能展示恢复动作。

### Phase 6. E2EE 配对完成
- 用 `e2ee-device-pairing` 完成三端安全边界对接。
- 固定安全链路：
  - `agentd` 首次运行生成长期密钥对。
  - 浏览器设备首次配对生成自己的长期密钥对。
  - `agentd auth` 创建一次性 token。
  - 手机端使用 token 发起配对并交换公钥。
  - relay 完成最小注册并返回访问能力。
- 三端边界固定：
  - `agentd/src/security/` 管桌面端密钥、token 生成、加密入口。
  - `relay/src/security/` 与 `src/storage/repositories/` 管 token 校验、设备注册、访问控制和密文存储边界。
  - `web/src/lib/crypto/` 与 `web/src/lib/storage/` 管浏览器密钥和配对设备持久化。
- 关键约束固定：
  - relay 不读取明文。
  - 未配对设备不得订阅。
  - 被撤销设备不得继续访问后续会话。
  - 私钥、真实 token、真实敏感数据不得进入仓库、日志、测试快照或截图。

### Phase 7. Windows 本地控制恢复完成
- 用 `windows-desktop-control` 完成 `apps/agentd/src/platform/windows/`。
- 固定平台职责：
  - 检测桌面端重新接管当前会话的明确本地输入事件。
  - 统一处理“显式把控制权还给桌面端”和“本地键盘恢复输入”两类触发。
  - 向会话层和事件发布层发出远控失效通知。
- 平台层只暴露稳定边界，不直接修改 transport、provider 或 web 逻辑。
- 成功标准：
  - 一旦桌面端恢复本地输入，远程控制命令立即失效。
  - 手机端能收到状态变更。
  - 当前会话状态与远控能力保持一致。

### Phase 8. 自托管部署完成
- 用 `docker-compose-selfhost` 补齐最小部署层。
- 固定部署文件：`compose.yaml`、`.env.example`、`infra/docker/relay.Dockerfile`。
- Compose 范围固定只有 `relay` 一个服务，SQLite 作为嵌入式存储，不引入 Redis/Postgres。
- 必须具备：
  - 端口映射。
  - 数据卷挂载。
  - 最小健康检查。
  - 环境变量注入。
  - 容器重启策略。
- 部署边界固定：
  - 服务于本地联调和单机自托管。
  - 不扩展到 TLS、反向代理、公网发布、多节点或云模板。

### Phase 9. 测试与验收完成
- 用 `qa-realtime-testing` 补齐各 workspace 测试骨架并逐步替换为真实测试。
- 测试分层固定：
  - `packages/protocol`：schema 与兼容性测试。
  - `apps/relay`：单测 + WebSocket/配对/回放/去重集成测试。
  - `apps/agentd`：单测 + 状态机/事件流/命令幂等/本地控制恢复集成测试。
  - `apps/web`：浏览器侧单测 + PWA 主链路冒烟测试。
- 根质量门槛固定：
  - `lint` 通过。
  - `typecheck` 通过。
  - 相关 workspace 测试通过。
  - 高风险改动必须附带回归测试或回归记录。

## Public Interfaces And Contracts
- 共享类型、schema、状态枚举、错误码只从 `packages/protocol` 导出。
- `agentd` 对 relay 的事件与命令收发边界必须以协议包为唯一源，不允许本地私有 wire shape。
- `web` 与 `relay` 的实时链路只通过 WebSocket 协议层通信，不直接耦合 relay 内部实现。
- `relay` 的配对、注册、订阅、回放、去重、设备撤销都必须以清晰的边界模块暴露，不允许散在 handler 内部。
- `compose.yaml`、`.env.example`、`relay.Dockerfile` 形成唯一部署入口，不允许各 workspace 各自藏部署脚本。

## Test Plan
- Monorepo 层：
  - workspace 能统一安装、构建、lint、typecheck、test。
- 协议层：
  - schema 校验通过。
  - 兼容性测试覆盖 `messageId`、状态枚举和关键消息类型。
- 配对与安全：
  - 有效 token 可配对。
  - 过期 token、已使用 token、撤销设备被拒绝。
  - relay 不能读取明文。
  - 错误设备密钥无法读取历史。
- 实时链路：
  - 启动 `codex` 后输出实时到达 web。
  - 断线后可重连并回放最近窗口事件。
  - 重复 `messageId` 不触发重复执行。
- 交互链路：
  - prompt 只执行一次。
  - 授权请求能展示并回传 allow/deny。
  - 显式释放控制权可生效。
  - 桌面本地输入恢复后远控立即失效。
- 部署层：
  - `docker compose config` 成功。
  - Compose 可在单机环境启动 relay。
  - SQLite 数据卷路径与环境变量配置正确。

## Delivery Sequence
- 第 1 周：完成 monorepo 基础层和协议层。
- 第 2 周：完成 relay 基础服务、SQLite 存储边界和最小配对 HTTP/WS 入口。
- 第 3 周：完成 agentd PTY、会话注册表、输出事件映射和命令处理。
- 第 4 周：完成 web PWA 页面壳、实时订阅壳和主链路 UI。
- 第 5 周：完成 E2EE 配对、设备注册、密文转发和设备撤销。
- 第 6 周：完成 Windows 本地控制恢复和远控失效闭环。
- 第 7 周：完成 Docker Compose 自托管、环境变量、健康检查和联调脚本。
- 第 8 周：补齐自动化测试、回归测试和端到端验收，收口文档与发布准备。

## Assumptions And Defaults
- 当前仓库仍处于“文档 + skills + 模板”阶段，尚未进入主应用实现阶段。
- 本计划默认以现有 9 个 project skills 为唯一实施加速器，不再新增新的必需 skill。
- 默认优先完成 V1 主链路，不提前扩展多用户、多租户、跨平台桌面、云部署、生产级监控。
- 默认 Windows 是唯一桌面目标平台，手机端优先是 PWA，不开发原生 App。
- 默认所有后续实现先保边界清晰，再补细节丰富度；如果时间不足，先保证主链路可跑通与可验证。
