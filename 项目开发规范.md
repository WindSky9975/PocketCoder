# PocketCoder 项目开发规范

## 1. 目标与适用范围

本规范用于约束 PocketCoder 仓库内的设计、开发、测试、提交、评审和文档维护方式。它服务于当前 V1 目标，不追求大公司级流程完整度，但要求规则足够明确，能直接指导实现。

本规范适用于：
- 仓库根目录及所有 workspace
- `web`、`agentd`、`relay`、`protocol` 等正式包
- 设计文档、接口文档、测试文档和运维文档
- 使用 AI 或 Codex 参与的代码、文档和测试改动

规则优先级如下：
- 安全性高于开发速度
- 正确性高于便利性
- 可维护性高于局部技巧
- 共享契约高于局部实现

核心设计原则如下：
- 高内聚：模块内部应围绕单一职责和单一变化原因组织代码，同一类问题优先在同一模块内闭环。
- 低耦合：模块之间只能通过稳定契约协作，禁止依赖其他模块的内部实现、目录细节或隐式状态。
- 先收敛边界，再扩展能力；新增功能优先落在已有高内聚模块内，只有现有模块无法承载时才新增模块或目录。

## 2. 仓库与目录约定

项目采用 `npm workspaces` 管理 monorepo，所有共享脚本从仓库根目录执行。

首版目录约定如下：
- `apps/web`：移动端优先的 Next.js PWA
- `apps/agentd`：Windows 优先的 Node.js CLI 守护进程
- `apps/relay`：Fastify + WebSocket 服务
- `packages/protocol`：共享类型、Zod schema、错误码和协议常量
- `设计方案/`：产品方案、实施规划和设计决策文档

目录与依赖约束如下：
- `packages/protocol` 是唯一跨端共享契约源，其他包不得复制消息结构、状态枚举或 schema。
- `web`、`agentd`、`relay` 只能依赖 `protocol`，不得直接互相引用源代码。
- 根目录统一放置 `tsconfig`、lint、format、测试和通用脚本配置。
- 新增 workspace 前必须说明职责、输入输出、依赖方向和是否属于长期维护模块。
- 新增 workspace 或顶级目录前，必须先说明为何现有模块无法承载该职责。
- 只有当逻辑稳定、跨端通用、边界清晰时，才允许抽取共享模块；禁止为了少量复用过早抽象。
- 禁止循环依赖、跨层直接调用，以及页面、CLI 入口直接访问底层存储或传输实现。
- 对外依赖必须指向稳定入口，禁止依赖其他模块的内部文件路径或未承诺导出的实现细节。
- 临时脚本、调研输出和调试文件不得长期混入正式目录；需要保留时应放入明确命名的工具或文档目录。

包命名与文件命名约定如下：
- npm 包名统一采用 `@pocketcoder/<name>`。
- 源码目录与包目录使用小写短横线或约定名称，避免随意缩写。
- 类型、接口、类、枚举、协议字段统一使用英文命名。

## 3. 编码与类型规范

默认技术约束如下：
- 所有正式运行时代码默认使用 TypeScript。
- TypeScript 必须启用严格模式，不允许关闭 `strict` 作为临时方案。
- 跨进程、跨网络、跨存储边界的数据必须显式校验，首选 Zod。
- 协议、权限、加密、会话状态等关键逻辑禁止依赖隐式约定。

代码风格要求如下：
- 代码、标识符、接口、协议字段、提交类型统一使用英文。
- 文档、注释、设计说明、PR 描述默认中文；必要术语可保留英文。
- 优先写清晰代码，避免为节省几行代码引入难读写法。
- 注释只解释意图、边界条件或非直观设计，不写重复代码表面的注释。
- 禁止滥用 `any`、类型断言链和绕过类型系统的快捷写法。
- 错误处理必须保留上下文；禁止静默吞错。

模块边界要求如下：
- 一个模块只负责一类清晰职责，并围绕一个核心场景或一类变化原因组织代码，不把协议定义、业务逻辑、存储逻辑、UI 展示混在同一层。
- 所有公共导出都应最小化，只暴露调用方真正需要的能力，避免让内部实现细节自然扩散成公共 API。
- 依赖方向必须单向；底层模块不得反向依赖上层模块，公共模块不得感知具体应用实现。
- 任何跨包复用都应先考虑放入 `protocol` 或明确的共享模块，而不是复制实现；但在复用语义尚未稳定前，不得为了“看起来更通用”而提前抽象。
- 如果一个改动需要同时修改多个无关模块，必须先评估是否存在职责分散、共享边界错误或耦合过高问题。
- 临时适配代码必须有明确边界和移除计划，不得自然扩散为长期公共接口。

日志与可观测性要求如下：
- 日志必须可读、可定位、可过滤。
- 日志中禁止输出密钥、token、明文 payload、设备私钥和真实用户敏感内容。
- 会话、设备、消息应使用稳定标识符记录，便于排查重连、幂等和授权问题。

## 4. 测试与质量门槛

仓库根目录必须提供统一脚本：
- `npm run dev`
- `npm run build`
- `npm run lint`
- `npm run typecheck`
- `npm run test`

质量门槛如下：
- 未通过 `lint`、`typecheck`、相关测试的改动不得合入。
- `protocol`、`relay`、`agentd` 的核心逻辑必须有自动化测试。
- `web` 首版至少覆盖关键页面和主链路冒烟测试。
- 涉及协议变更时，必须同步更新 schema、类型、测试和相关文档。
- 涉及配对、加密、权限、幂等、中断、重连、状态机的改动时，必须补充回归测试。
- 新增功能若跨越多个模块，必须在提交说明或设计文档中写明职责分配和依赖方向。
- 以解耦、重构或拆模块为目标的改动，必须提供最小回归测试或验证记录，证明行为未发生非预期变化。

测试分层要求如下：
- 单元测试：纯函数、schema、状态转换、去重逻辑、权限判断
- 集成测试：WebSocket 消息流、配对流程、会话订阅、事件回放
- 冒烟测试：核心页面可访问、主链路可操作、关键按钮和错误状态可见

最低验收标准如下：
- 新功能必须有明确验收路径，不能只靠“手动看起来能用”。
- Bug 修复优先补回归测试，再提交修复代码。
- 无法自动化覆盖的风险点，必须在提交说明或文档中写清楚验证方式。
- 验证应优先覆盖模块边界、公共接口和依赖关系的稳定性，而不是只验证内部实现细节。

## 5. Git 工作流

项目采用 `trunk-based` 开发：
- `main` 是唯一长期主干分支。
- 日常开发使用短生命周期分支，命名建议为 `feat/*`、`fix/*`、`docs/*`、`chore/*`。
- 一个分支只解决一个明确问题，避免把无关修改混进同一次提交。

提交规范采用 `Conventional Commits`：
- 格式：`type(scope): subject`
- 常用类型：`feat`、`fix`、`docs`、`refactor`、`test`、`chore`、`build`、`ci`
- `subject` 使用英文短句，聚焦结果，不写模糊描述

示例：
- `feat(protocol): add session subscribe schema`
- `fix(agentd): prevent duplicate prompt execution`
- `docs(repo): add development policy`

协作要求如下：
- 合并前先同步主干，减少大范围冲突。
- 不得在未说明原因的情况下混入重构、格式化、依赖升级等无关修改。
- 涉及公共接口、协议、配置项、目录结构的改动，必须在描述中明确影响面。
- 涉及结构性改动时，必须说明为何这样分层、当前耦合点在哪里、如何限制影响面。
- 新增共享模块、公共接口或目录层级时，必须说明它解决了什么具体复用问题或耦合问题。
- 纯文档修订可以简化流程，但仍需保持提交信息可读。

## 6. 配置与安全规范

配置管理要求如下：
- 所有环境变量必须有明确名称、用途、默认值策略和敏感级别。
- 可公开配置与敏感配置必须分开管理。
- 仓库内只提交示例配置，不提交真实凭据。
- 新增配置项时，必须同步更新说明文档。

密钥与凭据要求如下：
- 设备私钥、配对 token、访问密钥、第三方凭据不得进入仓库、日志、测试快照或截图。
- 本地开发如需生成密钥，必须写入本地安全存储或被忽略的配置文件。
- 任何打印到终端或写入日志的敏感字段都必须脱敏。

安全评审要求如下：
- 涉及配对、E2EE、权限控制、会话恢复、命令执行边界的改动，必须附带安全影响说明。
- 任何降低安全边界的“临时方案”都必须明确有效期和移除计划。
- 未经明确设计，不允许为了开发便利关闭鉴权、跳过校验或绕过幂等保护。

依赖与供应链要求如下：
- 新增依赖前先确认用途、维护状态和必要性。
- 能用现有依赖解决的问题，不新增重量级替代库。
- 与加密、PTY、进程管理、网络通信相关的依赖必须特别审查。

## 7. AI / Codex 协作规范

AI 可以用于生成草稿、补测试、整理文档、搭框架，但不能替代工程判断。

使用 AI 或 Codex 时必须遵守以下规则：
- 不向模型输入真实密钥、生产凭据、未脱敏日志或用户敏感数据。
- AI 生成的代码必须由人类阅读、运行和验证后才能提交。
- 协议、安全、并发、加密、进程控制相关代码必须做人类复核。
- 不得伪造测试结果、运行记录或“已验证”结论。
- 不得覆盖、回退或忽略他人未确认的改动。

AI 参与后的落地要求如下：
- 重要架构决策必须沉淀到仓库文档，不能只存在聊天记录里。
- 如果 AI 影响了协议、目录结构、公共接口或安全模型，必须同步更新相关文档。
- 当 AI 输出存在明显不确定性时，应优先补测试和验证，而不是直接接受实现。

## 8. 文档维护规则

文档不是附属品，而是项目的一部分。以下情况必须更新文档：
- 新增或删除 workspace
- 修改协议消息、状态机或错误码
- 修改配对流程、加密模型或权限边界
- 修改模块职责、依赖方向或共享边界
- 修改根脚本、环境变量、部署方式或测试入口
- 变更项目目标、范围或非兼容行为

文档维护要求如下：
- 方案文档记录“为什么这样做”。
- 实施规划记录“准备如何做”。
- 开发规范记录“必须怎样做”。
- 三类文档之间出现冲突时，必须在同一轮改动中一起修正。
- 当模块职责、依赖方向、共享边界发生变化时，必须同步更新 `设计方案/V1-项目结构.md` 或相关设计文档。

本规范默认由仓库维护者持续更新。任何偏离本规范的实现，都必须在提交说明或设计文档中明确解释原因。
