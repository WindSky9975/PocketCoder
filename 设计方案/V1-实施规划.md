# PocketCoder V1 实施规划

## Summary
基于现有方案文档，V1 按“真实可用最小版”落地，不做原型或模拟器。最终形态是一个单用户、自托管、Windows 优先的远程控制系统：手机端使用 Next.js PWA，桌面端使用 Node.js CLI 守护进程 `agentd`，服务端使用 Docker Compose 部署的 `relay + SQLite`，并从第一版开始实现设备配对和消息级 E2EE。

首阶段目标不是做全平台或完整产品包装，而是把以下主链路一次跑通：桌面启动 `codex` 会话、手机完成配对、手机查看实时输出、手机发送 prompt、手机处理授权、桌面输入恢复后撤销远控。

## Key Changes
### 1. Monorepo 与包边界
- 建立 TypeScript monorepo，固定 4 个 workspace：`web`、`agentd`、`relay`、`protocol`。
- 根层统一提供 TypeScript 配置、lint/test/build 脚本、环境变量约定和开发命令。
- `protocol` 作为唯一共享契约层，所有跨端消息、状态枚举、schema、错误码都从这里导出，禁止在 `web`、`agentd`、`relay` 各自重复定义。

### 2. `protocol` 契约
- 定义统一 envelope：`protocolVersion`、`messageId`、`timestamp`、`type`、`payload`。
- 首版固定支持的消息类型：`PairingInit`、`PairingConfirm`、`DeviceRegistered`、`SessionSummary`、`SessionSubscribe`、`SessionOutputDelta`、`SessionStateChanged`、`ApprovalRequested`、`ApprovalResponse`、`SendPrompt`、`InterruptSession`、`ResumeDesktopControl`、`Ack`、`ErrorEnvelope`。
- 定义统一会话状态：`idle`、`running`、`waiting_input`、`waiting_approval`、`error`、`disconnected`。
- 所有入站命令都要求 `messageId` 幂等；`relay` 和 `agentd` 都需要按 `messageId` 去重。
- 所有 schema 使用 Zod，并在 `relay` 和 `agentd` 入站边界强校验。

### 3. `agentd` 桌面端
- 形态固定为 Windows 优先的 Node.js CLI 守护进程，不做托盘，不做 GUI。
- 通过 PTY 启动真实 `codex` 进程，维护本地会话注册表，暴露统一 `sessionId/provider/status/currentTask/lastActivityAt` 模型。
- 负责解析 `codex` 输出并映射到协议事件，至少产出：`session.started`、`session.output.delta`、`session.state.changed`、`session.approval.requested`、`session.error`、`session.ended`。
- 接收入站命令：发送 prompt、授权响应、中断、恢复桌面控制、会话列表、会话订阅。
- 本地输入恢复检测按 Windows 优先实现：首版以“检测到桌面端重新接管当前会话的明确本地输入事件”为远控失效信号；一旦触发，立即广播状态变更并阻止远程继续发送控制命令。
- 本地长期设备密钥由 `agentd` 首次启动生成并持久化；提供 `agentd auth` 创建一次性配对 token，并输出 URL/二维码所需文本数据。

### 4. `relay` 服务端
- 技术路线固定为 Fastify + WebSocket + SQLite，自托管方式固定为 Docker Compose。
- `relay` 只承担设备注册、连接路由、加密消息转发、短期加密事件回放、心跳/在线状态，不运行任何 AI 会话逻辑。
- SQLite 存储最小化元数据：设备信息、配对 token、会话路由信息、心跳、TTL 加密事件 blob、已撤销设备标记、已处理命令去重记录。
- 配对流程固定为：`agentd auth` 生成一次性 token，手机端 PWA 使用 token 发起配对，双方交换公钥，`relay` 返回注册结果与最小访问能力。
- 消息内容默认端到端加密，`relay` 仅看到路由元数据和密文，不保留明文 payload。
- 断线重连时，`relay` 仅按会话和时间窗口回放最近的加密事件，不做历史长期归档。

### 5. `web` 移动端 PWA
- 技术路线固定为 Next.js App Router，移动端优先，首版即做 PWA 安装能力。
- 页面固定为：配对/登录页、会话列表页、会话详情页、授权操作面板、连接异常页。
- 会话详情页必须支持：实时输出流、底部固定 prompt 输入框、授权卡片 allow/deny、显式“把控制权还给桌面端”操作。
- 前端只负责设备密钥管理、消息加解密、会话展示与输入交互，不承担会话业务判断。
- WebSocket 为唯一主实时通道；页面刷新后通过已配对设备身份重新连接并订阅会话。

## Test Plan
- 配对链路：
  - 新桌面设备首次生成长期密钥和一次性 token。
  - 手机端使用有效 token 成功配对并注册设备。
  - 过期 token、已使用 token、撤销设备全部被拒绝。
- 会话控制：
  - `agentd` 启动真实 `codex` 后，输出能实时推送到 PWA。
  - 手机端发送 prompt，`agentd` 只执行一次。
  - 授权请求能在手机端展示，allow/deny 能准确回传。
  - 桌面端恢复本地输入后，远程控制立即失效并更新状态。
- 可靠性：
  - 手机断线后重连可以回放最近窗口内的加密事件。
  - `relay` 重启后不破坏已存在的桌面会话事实来源。
  - 重复投递同一 `messageId` 不会触发重复命令执行。
- 安全性：
  - `relay` 无法解密会话明文。
  - 错误设备密钥无法读取历史。
  - 被撤销设备无法继续订阅或接收后续事件。
- 工程验收：
  - monorepo 可以一次性安装、构建、测试。
  - Docker Compose 可以在单机环境启动 `relay` 及其存储依赖。
  - Windows 开发机能本地跑通端到端主链路。

## Assumptions
- 首版用户模型固定为单用户，不做团队协作和多租户。
- 首版桌面系统固定为 Windows，macOS/Linux 不进入 V1 开发范围。
- 首版必须接真实 `codex`，不先做协议模拟器，不先做只读模式。
- 首版安全模型采用完整 E2EE 配对，不接受仅 token/TLS 的降级方案。
- 首版部署目标固定为单机自托管 Docker Compose；云模板和 SaaS 化不进入当前计划。
