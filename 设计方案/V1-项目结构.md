# PocketCoder V1 项目结构

## 1. 结构目标

本结构基于以下约束整理：
- 采用 `npm workspaces` 管理 monorepo
- 正式 workspace 固定为 `web`、`agentd`、`relay`、`protocol`
- `packages/protocol` 是唯一跨端共享契约源
- 文档默认中文，代码与协议命名默认英文
- 测试按 workspace 就近放置，核心链路必须可自动化验证

该结构的目标不是一次性铺满所有未来能力，而是让 V1 可以直接启动开发，同时为后续接入其他 provider 保留扩展点。

## 3. 各目录职责

### 3.1 根目录
- `apps/` 放运行时应用，不放共享契约。
- `packages/` 放共享包，V1 仅保留 `protocol`。
- `infra/` 放部署相关文件，不混入业务代码。
- `.github/workflows/` 放 CI 流程，至少覆盖 `lint`、`typecheck`、`test`。
- `设计方案/` 放产品方案、实施规划、结构设计和后续 ADR。

### 3.2 `apps/web`
- `src/app/` 只放路由入口、页面级布局和页面装配逻辑。
- `src/features/` 按业务场景拆分功能模块，避免页面直接堆业务逻辑。
- `src/components/ui/` 仅放可复用基础组件，不放页面特定状态处理。
- `src/lib/crypto/` 负责浏览器端密钥管理和消息加解密。
- `src/lib/realtime/` 负责 WebSocket 连接、重连、订阅和事件分发。
- `src/lib/storage/` 负责本地设备信息、会话缓存和轻量持久化。
- `src/tests/` 放前端单测和主链路冒烟测试。

### 3.3 `apps/agentd`
- `src/cli/` 只处理命令入口与参数解析。
- `src/providers/codex/` 封装真实 `codex` 的 PTY 启动、输出解析和 provider 适配。
- `src/sessions/` 维护本地会话注册表、状态机和事实来源。
- `src/transport/` 处理与 relay 的连接、命令接收和事件上报。
- `src/security/` 处理设备密钥、配对逻辑和消息加密。
- `src/platform/windows/` 放 Windows 特有能力，例如桌面输入恢复检测。
- `tests/` 分离单测和集成测试，集成测试重点验证 PTY 包装、命令幂等和事件流。

### 3.4 `apps/relay`
- `src/modules/` 以领域模块划分服务端能力，不按控制器堆文件。
- `src/transport/http/` 负责配对、健康检查等 HTTP 接口。
- `src/transport/ws/` 负责实时连接、订阅、心跳和消息转发。
- `src/security/` 负责 token、设备注册和访问控制，不处理明文业务内容。
- `src/storage/migrations/` 管理 SQLite 表结构演进。
- `src/storage/repositories/` 统一封装设备、配对、会话路由、重放事件和去重记录访问。

### 3.5 `packages/protocol`
- `src/constants/` 放协议版本、消息类型、状态枚举。
- `src/schemas/` 放所有 Zod schema，是跨端消息的单一真相源。
- `src/errors/` 放标准错误码和错误 envelope 定义。
- `src/utils/` 放消息 ID 等与协议紧密相关的轻量工具。
- `tests/compatibility/` 用于保护协议兼容性，避免无意破坏消息格式。

## 4. 根级关键文件

- `package.json`：定义 workspace、根脚本和共享开发依赖。
- `package-lock.json`：锁定依赖版本，避免不同环境解析结果漂移。
- `tsconfig.base.json`：统一 TypeScript 严格模式和路径约束。
- `eslint.config.mjs`：统一 lint 规则，覆盖 Node、Next.js 和测试代码。
- `.gitignore`：忽略本地密钥、环境变量、构建产物、测试输出和调试目录。
- `README.md`：描述项目目标、开发启动方式和文档入口。

## 5. 结构约束

- V1 不新增第五个 workspace；公共能力优先收敛到 `protocol`，其余保持应用内聚。
- `web`、`agentd`、`relay` 不允许互相直接引用源码，只能通过 `protocol` 对齐契约。
- 测试默认跟随所属 workspace；不要额外建立独立测试 workspace。
- 临时调研目录、浏览器快照、控制台日志不属于正式项目结构。
- 当前仓库中的 `.playwright-mcp/` 应视为调研产物，不属于长期正式目录，后续应迁出或加入忽略规则。

## 6. 初始化顺序建议

1. 建立根目录 `package.json`、`tsconfig.base.json`、`eslint.config.mjs`、`.gitignore`。
2. 建立 `apps/web`、`apps/agentd`、`apps/relay`、`packages/protocol` 四个 workspace。
3. 先实现 `packages/protocol` 的 constants、schemas 和错误码，再让三个应用接入。
4. 再初始化 `relay` 和 `agentd` 的最小链路，最后补 `web` 页面与 PWA 能力。
5. 在主链路跑通后，再补充 CI、Dockerfile 和更完整的测试矩阵。
